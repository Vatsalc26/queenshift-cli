# Task Corpus

This file records the bounded task families Queenshift uses to talk about current capability honestly.

It is not a promise that every family is equally mature.

## Public-First Families

These are the most important families for the current public story:

1. `comment_file`
2. `update_named_file`
3. `bounded_two_file_update`
4. `update_file_and_test`
5. `rename_export`
6. `bounded_node_cli_task`

These families best match the current bounded coding thesis:

1. named targets
2. small explicit scope
3. visible review and replay
4. bounded verification surfaces

## Additional Evaluator And Broader-Coverage Families

These families exist in the broader project evidence, but they are not the first public defaults:

1. `create_tiny_file`
2. `sync_docs_with_source`
3. `sync_docs_bundle`
4. `medium_multi_file_update`
5. `cross_language_sync`

They matter for deeper evaluation and future growth, but they should not be mistaken for the calmest first-run story.

## How To Read This

1. Public docs should default to the public-first families.
2. Contributors can inspect the broader families, but the repo should not market them as equal-confidence defaults.
3. Evidence depth is asymmetric by design.

## Why This Matters

Queenshift is trying to stay respected by keeping capability claims proportional to proof.

That means:

1. first-run stories stay small and explicit
2. broader families remain visible without being oversold
3. the public repo stays aligned with the architecture note's "bounded first slice" message
